🏗️ Key Components of System Architecture

The CSIT Capstone Repository Management System is built upon five key architectural components, each serving a distinct role in the overall system design:

  • Hardware Platform — The hardware platform includes the physical components that support the system. The application runs on a local development server using XAMPP on Windows, which provides the Apache web server and MySQL database server. Local disk storage is used to securely store uploaded PDF manuscripts and file attachments. The network interface handles HTTP communication through localhost during development. The hardware platform was chosen to meet the system's requirements for file storage capacity and processing power needed for intensive PDF validation operations.
  • Software Platform — The system is built with PHP as the server-side programming language and utilizes the Laravel framework, which follows the Model-View-Controller (MVC) architectural pattern. MySQL serves as the relational database management system for structured data storage. Apache handles incoming HTTP requests, Blade templates coupled with Tailwind CSS render the responsive user interface, and SMTP services provide email delivery for system notifications.
  • System Interfaces — The public browse library allows guests and authenticated users to search and view published projects. Authentication pages securely handle user login and registration. Three role-specific dashboards — Student Dashboard, Adviser Dashboard, and Admin Panel — provide tailored interfaces for each user type. Finally, the project details page features an embedded PDF viewer for manuscript previews and attachment download capability.
  • System Structure — The system structure defines the overall organization and relationships between different components. Controllers handle incoming requests and coordinate business logic across project management, verification, and administration. Models define the data entities (User, Project, Author, Attachment, and ActivityLog) and their relationships through Eloquent ORM. Dedicated Services encapsulate reusable logic, such as the PDF Validator, email notifications, and duplicate hash checking.
  • Security — Security is enforced throughout the application to protect the system and users from unauthorized access. Authentication is handled through Laravel's session-based login with CSRF token protection. Authorization uses Role-Based Access Control (RBAC), strictly restricting system features based on the user's assigned role (Student, Adviser, or Admin). Passwords are encrypted using Bcrypt hashing and are never stored in plain text. Uploaded files undergo a rigorous multi-step validation process, including SHA-256 hash comparison for duplicate detection, ClamAV security scanning for file integrity, and PDF content verification. Finally, a comprehensive activity logging system maintains a complete audit trail of all significant system events.

👨‍🎓 Student Workflow

The student journey begins with account registration and ends with tracking the project status through the review pipeline:

  • Registration & Login: The student visits the public home page and fills out the registration form with their details, student ID, and program (BSInT or Com-Sci). The account is created securely but remains inactive until an administrator approves it. Once approved, the student logs into their personal dashboard.
  • Project Submission: The student submits a new project by providing required metadata (title, abstract, year, specialization, program), adding co-authors, selecting their adviser, uploading the manuscript PDF, and optionally attaching supplementary files. Submissions may be restricted by admin-configured deadlines or closure settings.
  • Automated Validation: Upon clicking submit, the system automatically runs three checks: SHA-256 duplicate hash detection, a ClamAV security scan of the uploaded file (including each attachment), and PDF content validation (checking for page counts and required keywords like the Approval Page). If any check fails, the submission is instantly blocked, all files are deleted, and the student receives a detailed error report.
  • Status Tracking & Cancellation: Passing all checks places the project in a "pending" state. The student can continuously monitor the progress from their dashboard or cancel the submission entirely as long as it has not yet been approved by the adviser.
  • Edit & Resubmit: If the adviser returns the project for revision (status: "rejected"), the student can edit the project metadata and resubmit it. The status is automatically reset to "pending" and the adviser is notified via email of the resubmission.

👨‍🏫 Adviser Workflow

The adviser role focuses on reviewing and verifying student-submitted manuscripts before they proceed to the admin for publication:

  • Notification & Review: The system sends an email alerting the adviser of a new pending submission. The adviser logs in, opens their verification queue, and reviews the manuscript alongside the system-generated automated validation report.
  • Decision Making: The adviser must make a decision:
    • Approve: Advances the project status to "approved," triggering automated email notifications to both the students and the system administrators.
    • Return: Reverts the project status to "rejected," requiring the adviser to provide a mandatory rejection reason, which is immediately emailed to the student so they can revise and resubmit.
  • Cancellation: If an approval was made in error, the adviser retains the ability to cancel their prior confirmation, dropping the project back to the "pending" state.

🛡️ Admin Workflow

The administrator has the broadest set of privileges, overseeing both user accounts and project publication:

  • User Management: The admin reviews pending student registration requests. They can approve accounts to grant login access, delete fraudulent requests, manually create new user accounts, or edit existing user details and privileges.
  • Project Publication: Once a project is marked "approved" by an adviser, it appears in the admin's publication queue. The admin performs a final review of the metadata and officially publishes the project.
  • Direct Upload of Past Projects: The admin can directly upload and publish past capstone projects without going through the student submission and adviser review pipeline. The admin fills a project form with all metadata, uploads the manuscript PDF and optional attachments, and the project is instantly published to the public library.
  • Category Management: The admin can create, edit, and delete specialization categories that are used to classify capstone projects. Categories in use by existing projects are protected from deletion.
  • System Settings: The admin can configure system-wide settings including maximum upload sizes, allowed file types, academic year, repository name, submission deadlines, and maintenance mode.
  • Action Logging & Notifications: Publishing changes the status to "published," making the research officially visible in the public library. Email notifications are automatically dispatched to the student authors and their adviser. Throughout this process, admins can monitor the application's comprehensive activity logs to audit all system actions.

🔄 Process Flow Description

The CSIT Capstone Repository Management System follows a five-phase linear pipeline to process each capstone project from submission to public archive:

  • Phase 1 — Submission: The flow begins when a student logs into their dashboard, completes the submission form, inputs all necessary metadata, and uploads their final manuscript PDF.
  • Phase 2 — Automated Verification: The system intercepts the upload and executes three sequential checks. First, it computes a SHA-256 hash to intercept duplicate uploads. Next, it runs a file integrity security scan. Lastly, it reads the PDF text to validate structure and required signatures. A failure at any step safely rolls back the submission. Success moves the project to a "pending" queue.
  • Phase 3 — Faculty Review: The newly pending project alerts the assigned faculty adviser. The adviser reads the manuscript through the system's built-in PDF viewer, reviews the automated validation report, and decides to either return it to the student for corrections or officially approve it.
  • Phase 4 — Admin Publication: An adviser's approval alerts the system administrator. The admin conducts a final metadata review and officially publishes the project.
  • Phase 5 — Archival & Retrieval: The successfully published project is permanently archived into the relational database. It is fully indexed, allowing any guest or user to perform deep, full-text searches across the public library utilizing advanced filters to retrieve documents by title, year, specialization, keyword, or author.

📊 System Flowchart Description

The detailed system flowchart above shows the complete end-to-end flow of a capstone project through the CSIT Repository, reading left-to-right. Each decision diamond branches downward for the failure/reject path, while the success path continues rightward:

  • Submission Entry Point — The flow begins when a student logs in and submits a project through the submission form, providing all required metadata and uploading the manuscript PDF.
  • Three-Stage Automated Verification — The system runs three sequential validation checks. Each check is a decision point that can either pass the file forward (→) or terminate the flow downward (↓) with an error:
    • ① SHA-256 Check — Computes a SHA-256 hash of the uploaded PDF and compares it against all existing manuscripts. If a match is found, the submission is blocked and rolled back.
    • ② ClamAV Scan — Runs a security scan. If the scan detects an issue, the file is deleted and the failure is logged.
    • ③ PDF Validation — Validates the PDF content by checking page count, document structure, and the presence of required section keywords. If validation fails, the file is deleted and a detailed report is shown.
  • Adviser Review Decision — After all automated checks pass, the project is created with "pending" status and the adviser receives an email notification. The adviser reviews the manuscript and either approves it (→ continues right to admin) or returns it for revision (↓ drops down, student revises and resubmits back to Step ①).
  • Admin Publication Decision — Once approved, the admin reviews the project and either publishes it (→ making it publicly visible with email notifications) or holds it (↓ keeping it in "approved" status for later).
  • Archival — Published projects become permanently available in the public browse library, fully indexed and instantly searchable by any user or guest via Full-Text Search.
