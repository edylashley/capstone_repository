🏗️ Key Components of System Architecture

The CSIT Capstone Repository Management System is built upon five key architectural components, each serving a distinct role in the overall system design:

  • Hardware Platform — The application runs on a local development server using XAMPP on Windows, which provides the Apache web server and MySQL database server. Local disk storage is used to securely store uploaded PDF manuscripts and file attachments. The network interface handles HTTP communication through localhost during development.
  • Software Platform — The system is built with PHP 8.2 as the server-side programming language and utilizes the Laravel 12 framework, which follows the Model-View-Controller (MVC) architectural pattern. MySQL serves as the relational database management system for structured data storage. Apache handles incoming HTTP requests, Blade templates coupled with Tailwind CSS render the responsive user interface, and Gmail SMTP services provide queued email delivery for system notifications.
  • System Interfaces — The public browse library allows guests and authenticated users to search and view published projects. Authentication pages handle user login, registration with email verification. Two role-specific dashboards — Student Dashboard and Admin Dashboard — provide tailored interfaces for each user type. The system includes an embedded PDF viewer for manuscript previews, and a support ticket system for user feedback and issue reporting.
  • System Structure — Controllers handle incoming requests and coordinate business logic across authentication, project management, user administration, support tickets, and archive management. Models define the data entities (User, Project, ProjectFile, Category, Program, SupportTicket, and ActivityLog) and their relationships through Eloquent ORM. Dedicated Services encapsulate reusable logic, including the PDF Validator for content checking and the File Scanner for ClamAV security scanning.
  • Security — Authentication is handled through Laravel's session-based login with CSRF token protection and a two-step email verification during registration. Authorization uses Role-Based Access Control (RBAC), restricting system features based on the user's assigned role (Student or Admin). Passwords are encrypted using Bcrypt hashing. Uploaded files undergo a multi-step validation process, including SHA-256 hash comparison for duplicate detection, ClamAV security scanning, and PDF content verification. A comprehensive activity logging system maintains a complete audit trail of all significant system events.

👨‍🎓 Student Workflow

The student journey begins with account registration and ends with tracking the project status through the review pipeline:

  • Registration & Email Verification: The student visits the public home page and fills out the registration form with their details, student ID, and program. The system sends a 6-digit verification code to the student's email address. The student must enter this code correctly within 10 minutes to confirm their email. Upon successful verification, the account is created but remains inactive until an administrator approves it.
  • Project Submission: The student submits a new project by providing required metadata (title, abstract, year, program, categories), adding authors, specifying their adviser name, uploading the manuscript PDF, and optionally attaching supplementary files. Submissions may be restricted by admin-configured deadlines or closure settings.
  • Automated Validation: Upon clicking submit, the system automatically runs three checks: SHA-256 duplicate hash detection, a ClamAV security scan of the uploaded file (including each attachment), and PDF content validation (checking for page counts and required keywords). If any check fails, the submission is instantly blocked, all files are deleted, and the student receives a detailed error report. Security threats automatically generate a high-priority support ticket for the admin.
  • Status Tracking & Cancellation: Passing all checks places the project in a "pending" state. The student can monitor the progress from their dashboard or cancel the submission entirely as long as it has not yet been published.
  • Edit & Resubmit: If the admin returns the project for revision (status: "returned"), the student can edit the project metadata and resubmit it. The status is automatically reset to "pending" and re-enters the review queue.

🛡️ Admin Workflow

The administrator has the broadest set of privileges, overseeing both user accounts and project publication:

  • User Management: The admin reviews pending student registration requests. They can approve accounts (triggering an email notification to the student), deny and soft-delete fraudulent requests, manually create new user accounts, or edit existing user details and privileges.
  • Project Review & Publication: When a student submits a project, the admin receives an email notification. The admin reviews the project metadata, manuscript PDF, and automated validation report. The admin can either approve and publish the project directly (making it publicly visible and sending email notifications to the authors) or return it for revision with a mandatory written reason (emailed to the student for correction).
  • Direct Upload of Past Projects: The admin can directly upload and publish past capstone projects without going through the student submission pipeline. This supports both single project upload and bulk upload (multiple projects in one submission). Projects are instantly published to the public library.
  • Category & Program Management: The admin can create, edit, and delete specialization categories and academic programs that are used to classify capstone projects. Items in use by existing projects are protected from deletion.
  • System Settings: The admin configures system-wide settings including maximum upload sizes, allowed file types, academic year, repository name, submission deadlines, maintenance mode, and log retention days.
  • Support Tickets: The admin can view, filter, search, reply to, and resolve user-submitted support tickets. Automated security alert tickets are generated when malicious uploads are detected. Resolved tickets auto-expire after 3 days.
  • Archive Center: The admin manages soft-deleted records (projects, users, categories) through a centralized trash interface, with options to restore or permanently purge items.
  • Activity Logs: The admin monitors a comprehensive audit trail of all significant system events, with configurable retention periods and automatic pruning.

🔄 Process Flow Description

The CSIT Capstone Repository Management System follows a four-phase linear pipeline to process each capstone project from submission to public archive:

  • Phase 1 — Submission: The flow begins when a student logs into their dashboard, completes the submission form, inputs all necessary metadata, and uploads their final manuscript PDF along with optional attachments.
  • Phase 2 — Automated Verification: The system intercepts the upload and executes three sequential checks. First, it computes a SHA-256 hash to intercept duplicate uploads. Next, it runs a ClamAV file integrity security scan. Lastly, it reads the PDF text to validate structure and required signatures. A failure at any step safely rolls back the submission and generates a security alert ticket for the administrator. Success moves the project to a "pending" queue.
  • Phase 3 — Administrative Review: The newly pending project alerts the system administrator via email. The admin reviews the manuscript through the system's built-in PDF viewer and the automated validation report, then decides to either return it to the student for corrections or officially approve and publish it directly to the public library.
  • Phase 4 — Publication & Archival: The successfully published project is permanently archived into the relational database. It is fully indexed, allowing any guest or user to perform deep, full-text searches across the public library utilizing advanced filters to retrieve documents by title, year, program, category, keyword, or author.

📊 System Flowchart Description

The detailed system flowchart above shows the complete end-to-end flow of a capstone project through the CSIT Repository, reading left-to-right. Each decision diamond branches downward for the failure/reject path, while the success path continues rightward:

  • Submission Entry Point — The flow begins when a student logs in and submits a project through the submission form, providing all required metadata and uploading the manuscript PDF.
  • Three-Stage Automated Verification — The system runs three sequential validation checks. Each check is a decision point that can either pass the file forward (→) or terminate the flow downward (↓) with an error:
    • ① SHA-256 Check — Computes a SHA-256 hash of the uploaded PDF and compares it against all existing manuscripts. If a match is found, the submission is blocked and rolled back.
    • ② ClamAV Scan — Runs a security scan. If the scan detects a threat, the file is deleted, the failure is logged, and an automated security support ticket is generated for the administrator.
    • ③ PDF Validation — Validates the PDF content by checking page count, document structure, and the presence of required section keywords. Results are logged in a validation report for admin review.
  • Admin Review Decision — After all automated checks pass, the project is created with "pending" status and the administrator receives an email notification. The admin reviews the manuscript and validation report, then either publishes it directly (→ making it publicly visible with email notifications to all authors) or returns it for revision (↓ student revises and resubmits back to Step ①).
  • Archival — Published projects become permanently available in the public browse library, fully indexed and instantly searchable by any user or guest via full-text search.
