🏗️ Key Components of System Architecture

The CSIT Capstone Repository Management System is built upon five key architectural components, each serving a distinct role in the overall system design:

  • Hardware Platform — The hardware platform includes the physical components that support the system. The application runs on a local development server using XAMPP on Windows, which provides the Apache web server and MySQL database server. Local disk storage is used to store uploaded PDF manuscripts and file attachments. The network interface handles HTTP communication through localhost during development. The hardware platform was chosen to meet the system's requirements for file storage capacity and processing power needed for PDF validation operations.
  • Software Platform — The software platform includes the operating system, application server, and software components that run on the hardware. The system is built with PHP 8.x as the server-side programming language and uses the Laravel framework, which follows the Model-View-Controller (MVC) architectural pattern. MySQL serves as the relational database management system for structured data storage. Apache handles incoming HTTP requests, Blade templates render the user interface, and Gmail SMTP provides email delivery for system notifications. These technologies were chosen based on their compatibility, community support, and suitability for web application development.
  • System Interfaces — System interfaces include the user interfaces used to interact with the system. The public browse library allows guests and authenticated users to search and view published projects. Authentication pages handle user login and registration. Three role-specific dashboards — Student Dashboard, Adviser Dashboard, and Admin Panel — provide tailored interfaces for each user type. The project details page features an embedded PDF viewer for manuscript preview and file attachment download capability. All interfaces are designed to be user-friendly and responsive, capable of handling the expected volume of concurrent users.
  • System Structure — The system structure defines the overall organization and the relationships between different components. Controllers handle incoming requests and coordinate business logic across authentication, project management, verification, and administration. Models define the data entities (User, Project, Author, Attachment, and ActivityLog) and their relationships through Eloquent ORM. Services encapsulate reusable business logic such as PDF validation, email notification, and duplicate hash checking. Routes define the URL endpoints and apply role-based middleware for access control. Database migrations maintain version-controlled schema definitions. This modular structure enables the system to be scalable, allowing new features and components to be added with minimal impact on existing functionality.
  • Security — Security is a critical aspect of the system architecture, designed to protect both the system and its users from unauthorized access. Authentication is enforced through Laravel's session-based login system with CSRF token protection. Authorization uses role-based access control, restricting system features based on the user's assigned role (Student, Adviser, or Admin). Passwords are encrypted using Bcrypt hashing and are never stored in plain text. Uploaded files undergo a multi-step validation process including SHA-256 hash comparison for duplicate detection, security scanning for file integrity, and PDF content verification. A comprehensive activity logging system maintains a complete audit trail of all significant system events for monitoring and accountability.

👨‍🎓 Student Workflow

The student journey begins with account registration and ends with tracking the project status through the review pipeline:

  • Registration — The student visits the public home page and fills out the registration form with their name, email, password, and student ID. The account is created in an inactive state, requiring administrator approval before the student can log in.
  • Login & Dashboard — Once approved, the student logs in and is redirected to their personal dashboard, where they can see submitted projects and their current statuses.
  • Project Submission — The student fills out the submission form with project metadata (title, abstract, year, specialization), adds co-authors, selects their adviser, uploads the manuscript PDF, and optionally attaches additional files.
  • Automated Validation — Upon submission, the system automatically runs three checks: SHA-256 duplicate hash detection against existing manuscripts, security scanning of the uploaded file, and PDF content validation (page count and keyword verification). If any check fails, the submission is blocked and the student is shown a detailed error report.
  • Status Tracking — After a successful submission, the project enters the "pending" state. The student can monitor progress from their dashboard and will receive email notifications when the adviser approves or returns the project.
  • Cancellation — Students may cancel their submission only while the project status is still "pending" or "rejected."

👨‍🏫 Adviser Workflow

The adviser role focuses on reviewing and verifying student-submitted manuscripts before they proceed to the admin for publication:

  • Notification — When a student submits a project and names the adviser, the system automatically sends an email notification to the adviser, alerting them of a new submission ready for review.
  • Dashboard Access — The adviser logs in and is directed to their dashboard, which displays all projects where they have been assigned as the adviser. A dedicated verification queue lists projects awaiting their review.
  • Manuscript Review — The adviser opens each pending project, reads the manuscript through the built-in PDF viewer, checks the metadata (title, abstract, authors), and reviews the automated validation report.
  • Re-validation — The adviser may choose to re-run the PDF validation at any time to obtain an updated report, especially if the validation criteria or the manuscript have been discussed with the student.
  • Approval or Return — After reviewing, the adviser either approves the project (advancing the status to "approved," which triggers email notifications to the students and administrators) or returns it for revision (setting the status to "rejected," which notifies the student via email).
  • Cancel Confirmation — If an approval was made in error, the adviser can cancel their prior confirmation, which reverts the project status back to "pending."

🛡️ Admin Workflow

The administrator has the broadest set of privileges, overseeing both user accounts and project publication:

  • User Account Management — The admin reviews pending registration requests from the User Accounts panel. Inactive accounts are listed first for easy action. The admin can approve accounts (setting them to active so the user can log in), delete accounts, edit user details (name, email, role, status), toggle active/inactive status, or create new users directly.
  • Project Publication — Once a project has been approved by an adviser, it appears in the admin's project flow filtered by "approved" status. The admin reviews the project metadata, optionally edits fields (title, year, adviser), and can re-run PDF verification if needed. The admin then decides to publish the project (setting its status to "published" and making it visible in the public browse library) or hold it for further review.
  • Email Notifications — When a project is published, the system automatically sends email notifications to the student authors and the assigned adviser, informing them of the publication.
  • Activity Monitoring — The admin has access to a comprehensive system activity log that tracks all significant events: project uploads, manuscript validations, approvals, rejections, publications, deletions, and account management actions.
  • Project Deletion — The admin may permanently delete a project and its associated files from the system if required.

🔄 Process Flow Description

The CSIT Capstone Repository Management System follows a five-phase linear pipeline to process each capstone project from submission to public archive:

  • Phase 1 — Submission: The student initiates the workflow by logging into their dashboard, completing the submission form with all required metadata (title, abstract, year, specialization, authors, and adviser), uploading the manuscript as a PDF, and optionally attaching supplementary files. This phase is entirely student-driven.
  • Phase 2 — Automated Verification: Immediately upon submission, the system triggers an automated three-step validation pipeline. The SHA-256 hash check ensures no duplicate manuscript exists in the repository. The security scan checks the file for malicious content. The PDF content validation verifies structure, page count, and required keywords. Projects that pass all checks are created with a "pending" status; failures result in immediate rejection with detailed error feedback.
  • Phase 3 — Faculty Review: The assigned adviser is notified via email and reviews the project through their verification queue. The adviser can read the manuscript in the built-in PDF viewer, inspect the automated validation report, and optionally re-run the PDF validation. The adviser then either approves the project (moving it forward to the admin) or returns it for revision (notifying the student to make corrections and resubmit).
  • Phase 4 — Admin Publication: Upon adviser approval, the administrator receives an email notification and reviews the project for final publication. The admin verifies metadata accuracy, optionally edits fields, and publishes the project. Publication sets the status to "published," making the project visible in the public browse library. All stakeholders (student authors and the adviser) are notified via email.
  • Phase 5 — Archival & Retrieval: Published projects are permanently archived in the centralized database and become fully accessible through the public browse library. Users can search and retrieve projects using multiple filters such as title, year, specialization, keyword, author, or adviser. The project details page provides a full PDF viewer and attachment download capability.

📊 System Flowchart Description

The detailed system flowchart above shows the complete end-to-end flow of a capstone project through the CSIT Repository, reading left-to-right. Each decision diamond branches downward for the failure/reject path, while the success path continues rightward:

  • Submission Entry Point — The flow begins when a student logs in and submits a project through the submission form, providing all required metadata and uploading the manuscript PDF.
  • Three-Stage Automated Verification — The system runs three sequential validation checks. Each check is a decision point that can either pass the file forward (→) or terminate the flow downward (↓) with an error:
    • ① Duplicate Check — Computes a SHA-256 hash of the uploaded PDF and compares it against all existing manuscripts. If a match is found, the submission is blocked and rolled back.
    • ② Security Scan — Runs a file integrity check. If the scan detects an issue, the file is deleted and the failure is logged.
    • ③ PDF Validation — Validates the PDF content by checking page count, document structure, and the presence of required section keywords. If validation fails, the file is deleted and a detailed report is shown.
  • Adviser Review Decision — After all automated checks pass, the project is created with "pending" status and the adviser receives an email notification. The adviser reviews the manuscript and either approves it (→ continues right to admin) or returns it for revision (↓ drops down, student revises and resubmits back to Step ①).
  • Admin Publication Decision — Once approved, the admin reviews the project and either publishes it (→ making it publicly visible with email notifications) or holds it (↓ keeping it in "approved" status for later).
  • Archival — Published projects become permanently available in the public browse library, fully indexed and searchable by any user or guest.
